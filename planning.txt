# Weaving Draft Web App — Planning & Design Document

---

## 1. Project Overview

### What Is a Weaving Draft?
A weaving draft is a set of four interconnected grids that together describe how a woven fabric is constructed on a shaft loom:

```
        ┌─────────────────────┬────────────┐
        │     THREADING       │  TIE-UP    │
        │  (warp × shafts)    │(shafts ×   │
        │                     │ treadles)  │
        ├─────────────────────┤            │
        │   WARP COLORS       │            │
        ├─────────────────────┼────────────┤
        │     DRAWDOWN        │ TREADLING  │
        │  (warp × weft)      │(weft ×     │
        │  [auto-computed]    │ treadles)  │
        └─────────────────────┴────────────┘
   WEFT COLORS
   (shown on left edge of drawdown)
```

| Grid | What It Represents |
|---|---|
| **Threading** | Which shaft each warp thread passes through |
| **Tie-up** | Which shafts each treadle raises |
| **Treadling** | Which treadle is pressed for each weft row |
| **Drawdown** | The resulting fabric pattern *(computed automatically)* |

### Goal
Build a single-page, client-side web app hosted on GitHub Pages that allows a user to interactively create, edit, visualize, save, and export weaving drafts — and track their physical progress at the loom during both setup (threading) and weaving (treadling).

### Design Principles
- **Local-first** — everything works offline, data stays on the user's machine
- **Simple to start, powerful when needed** — sensible defaults, progressive complexity
- **Loom-side companion** — progress tracking makes the app useful *while* weaving, not just during design
- **Standard formats** — support WIF import/export for interoperability with other weaving software
- **Future-portable** — clean architecture that allows adding a backend later without rewriting

## 3. Data Model

### Draft Schema

```
interface Draft {
  id: string;               // UUID (crypto.randomUUID)
  name: string;
  createdAt: string;        // ISO 8601
  updatedAt: string;        // ISO 8601
  loom: LoomSettings;
  threading: number[];      // length = warpCount, values = shaft index (0-based)
  tieup: boolean[][];       // [shaft][treadle]
  treadling: number[];      // length = weftCount, values = treadle index (0-based)
  warpColors: string[];     // length = warpCount, hex color strings
  weftColors: string[];     // length = weftCount, hex color strings
  palette: string[];        // saved color palette for this draft
  progress: ProgressState;
}

interface LoomSettings {
  shafts: number;           // 2–32
  treadles: number;         // 2–32
  warpCount: number;        // number of warp ends
  weftCount: number;        // number of weft picks
}

interface ProgressState {
  currentThread: number;    // 0-based index, -1 = not started
  currentPick: number;      // 0-based index, -1 = not started
}
```

### Default Values

| Setting | Default |
|---|---|
| Shafts | 4 |
| Treadles | 6 |
| Warp ends | 60 |
| Weft picks | 60 |
| Threading | Straight draw (0-1-2-3-0-1-2-3...) |
| Tie-up | Twill + tabby (see `createDefaultTieup` in draft.js) |
| Treadling | Treadles 3-4-5-6 repeating (0-indexed: 2,3,4,5) |
| Warp color | Black (`#000000`) |
| Weft color | White (`#ffffff`) |
| Palette | Black, white, red, blue, green, orange |
| Progress | Both at -1 (not started) |

The tie-up is defined using `tieupFromMap()` which takes a readable `{ treadle: [shaft, ...] }` mapping.

### Drawdown Computation (Pure Function)

```
For each cell (warpIndex, weftIndex):
  shaft    = threading[warpIndex]
  treadle  = treadling[weftIndex]
  warpOnTop = tieup[shaft][treadle]

  if warpOnTop → show warpColors[warpIndex]
  else         → show weftColors[weftIndex]
```

Deterministic, derived entirely from the three input grids + colors. Never stored — always recomputed. O(warp × weft), runs in < 1ms for 500×500.

## 4. Architecture

```
┌─────────────────────────────────────────────────┐
│                    UI Layer                      │
│  ┌───────────┐ ┌───────────┐ ┌───────────────┐  │
│  │ Toolbar / │ │ Grid      │ │ Color/Pattern │  │
│  │ Controls  │ │ Canvases  │ │ Tools         │  │
│  └───────────┘ └───────────┘ └───────────────┘  │
├─────────────────────────────────────────────────┤
│                 App State                        │
│  Single source of truth (Draft object)           │
│  Direct mutation → re-render affected grids      │
├──────────────┬──────────────┬───────────────────┤
│ Draft Engine │ Storage Svc  │ File I/O Service  │
│ (compute     │ (save/load)  │ (WIF, JSON, PNG)  │
│  drawdown)   │              │                   │
└──────────────┴──────────────┴───────────────────┘
```

### Actual Module Breakdown (as built)

```
index.html                  # Root (for GitHub Pages), refs into src/
src/
├── app.js                  # Entry point, state, event wiring
├── style.css               # All styles (CSS custom properties for theming)
├── model/
│   ├── draft.js            # Draft creation, validation, defaults, tieupFromMap
│   └── drawdown.js         # Pure drawdown computation
├── services/
│   └── storage.js          # localStorage save/load (async interface)
└── ui/
    ├── gridCanvas.js       # Reusable canvas grid: render, scroll, click, drag-to-paint
    ├── palette.js           # Color palette panel (active swatch + swatches)
    ├── colorTools.js        # Fill All + Repeat Pattern builder for colors
    └── sequenceTools.js     # Repeat Pattern builder for threading/treadling (uses mini GridCanvas)
```

Note: `index.html` lives at the repo root (GitHub Pages requirement) and references `src/` files. No build step.

### Technology Choices

| Concern | Choice | Rationale |
|---|---|---|
| **Framework** | Vanilla JS (ES modules) | Zero build step, runs directly on GH Pages, no dependencies |
| **Grid rendering** | HTML `<canvas>` | Performant for large grids (1000+ cells); pixel-level control |
| **State management** | Single `Draft` object, direct mutation | Simple; no library needed for a single-user app |
| **Styling** | Plain CSS with custom properties | Theming via variables, no tooling |
| **Build step** | None | Start simple; add bundling only if needed |

## 5. UI Layout

### Current Desktop Layout

```
┌──────────────────────────────────────────────────────────────────┐
│  Weaving Draft   [draft ▼] [name____] [New] [Save] [Delete]     │
├──────────────┬───────────────────────────────────────────────────┤
│              │                                                   │
│  Palette:    │            THREADING       │  TIE-UP              │
│  [■] [swatches]           [Pattern...]    │                      │
│              │        ────────────────    │                      │
│  Weft Colors:│         WARP COLOR BAR                            │
│  [Fill All]  │        ────────────────────┬──────────            │
│  [Repeat...] │  Weft  │   DRAWDOWN        │ TREADLING            │
│              │  Color │  (auto-computed)   │ [Pattern...]         │
│              │  Bar   │                   │                      │
│              │        │                   │                      │
├──────────────┴────────┴───────────────────┴──────────────────────┤
│  4 shafts · 6 treadles · 60 ends · 60 picks                     │
└──────────────────────────────────────────────────────────────────┘
```

Key layout decisions:
- **Warp color bar** sits between threading and drawdown (above drawdown)
- **Weft color bar** sits to the left of drawdown, aligned with the drawdown canvas
- **Palette** in left sidebar — always visible
- **Color tools** (Fill All + Repeat Pattern) in sidebar for weft, above grids for warp
- **Pattern tools** for threading/treadling are inline buttons that expand a mini-grid builder

### Interaction Model

| Action | Behavior |
|---|---|
| **Click cell** in threading | Sets that warp end to that shaft (exclusive per column) |
| **Click cell** in tie-up | Toggles connection (multiple per column allowed) |
| **Click cell** in treadling | Sets that weft pick to that treadle (exclusive per row) |
| **Click/drag** warp color bar | Paints cells with active palette color |
| **Click/drag** weft color bar | Paints cells with active palette color |
| **Wheel** over any grid | Scrolls; synced grids follow |
| **Click draft name** | Edits name inline |

### GridCanvas Component

The `GridCanvas` class is the core reusable component. Each grid is an instance. It handles:

- Virtualized rendering (only visible cells drawn)
- Viewport clipping with configurable max width/height
- HiDPI-aware canvas sizing
- Hit-testing (pixel → cell coordinates, accounting for scroll)
- Hover highlight
- Click and drag-to-paint (mousedown + mousemove fires onCellClick per cell)
- Scroll via wheel events
- Synchronized scrolling via `setScrollX`/`setScrollY` + `onScroll` callback

```
Cell size: 20px (fixed for now)
Grid lines: 0.5px, #cccccc
Filled cells: solid color (monochrome) or per-cell color
Hover: blue translucent highlight + border
```

## 6. Progress Tracking (NOT YET IMPLEMENTED)

### Overview

Two sequential phases a weaver performs:

| Phase | Task | Tracks | Direction |
|---|---|---|---|
| **Setup** | Threading warp through heddles | `currentThread` — which warp end | Left → right |
| **Weaving** | Throwing each pick | `currentPick` — which weft row | Top → bottom |

### Visual Design

- **Threading progress**: vertical highlight band (blue) on threading, drawdown, and warp color bar
- **Treadling progress**: horizontal highlight band (gold) on treadling, drawdown, and weft color bar
- When both active: crosshair on drawdown

### Progress Panel

Collapsible sidebar panel with:
- Current position / total (e.g., "End 87 / 120")
- Progress bar
- Prev / Next buttons
- Reset button

### Keyboard Shortcuts

| Key | Action |
|---|---|
| `→` | Next warp end |
| `←` | Previous warp end |
| `↓` or `Space` | Next pick |
| `↑` | Previous pick |

Only active when no text input is focused.

### Auto-Scroll

Advancing position should keep the highlight centered in the viewport. Progress auto-saves with every advance.

## 7. Storage Service

Implemented in `services/storage.js`. Async interface (localStorage now, API later):

- `listDrafts()` → `[{ id, name, updatedAt }]` sorted by most recent
- `loadDraft(id)` → full Draft object or null
- `saveDraft(draft)` → upserts by id, sets updatedAt
- `deleteDraft(id)` → removes by id

All drafts stored under a single `weaving-drafts` localStorage key. Auto-save fires on every edit. localStorage (5–10 MB) holds 1,000+ drafts easily.

## 10. Known Bugs

- **Weft color bar alignment**: the `&nbsp;` label spacer for vertical alignment is fragile — may break if label font/size changes
- **Drag-to-paint on grids other than color bars**: mouseup listener is on `window`, so dragging that starts on one grid and ends on another may leave stale drag state
- **No loom configuration UI yet**: shafts, treadles, warp count, weft count are hardcoded defaults — no way to change them after draft creation
- **Pattern tool mini-grid**: when adding/removing slots, the entire builder DOM is rebuilt which can feel janky
- **Color bar scroll sync**: scrolling very fast can cause slight desync between grids (no rAF batching)
- **No input validation on draft load**: corrupted localStorage data will crash the app

## 11. Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Canvas performance on very large drafts (1000+ threads) | Low | Already virtualized — only visible cells rendered |
| localStorage data loss | Medium | JSON download/upload planned for Phase 2. Consider auto-export reminder |
| WIF format edge cases | Medium | Test against files from multiple programs. Be lenient on import |
| Scope creep | High | Stick to MVP. Ship Phase 1 before Phase 2 |
| Mobile usability | Medium | Desktop is primary target. Don't break on mobile but don't optimize for it |

## 12. Open Questions

- **Multi-treadle per pick?** Some weaving styles press multiple treadles simultaneously. DEFER.
- **Liftplan mode?** Alternative to tie-up + treadling for table/dobby looms. DEFER.
- **Threading direction?** Some weavers thread right-to-left. Progress tracker should support both. YES — implement when building progress tracking.
- **Yarn-level simulation?** Thread crossings instead of solid blocks. Phase 3+.
- **Progress notifications?** "You're 50% through threading!" YES — implement as part of progress panel.
